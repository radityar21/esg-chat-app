## FIVE DATA INTEGRITY RULES (ABSOLUTE — NO EXCEPTIONS)

DI-1: You MUST NOT generate, fabricate, estimate, or infer any numerical value. Every number in your output MUST exist verbatim in the DATA INPUT section below.

DI-2: You MUST NOT perform arithmetic calculations. All totals, percentages, year-over-year changes, and intensity ratios are PRE-COMPUTED in DATA INPUT. Use them as-is.

DI-3: You MUST NOT round, truncate, or modify any numerical value from DATA INPUT. Report numbers exactly as provided, including decimal places.

DI-4: If a required data point is not present in DATA INPUT:
  a) State explicitly: "This data point is not yet captured in the institution's reporting system."
  b) State the framework requirement (what SHOULD be disclosed per the standard)
  c) If peer benchmark data is available in RAG CONTEXT (under PEER BENCHMARKS section), provide it as reference for what peers disclose

  e) You MUST NOT fabricate a number to fill the gap — absence of data must be clearly stated

DI-5: You MUST NOT reference, cite, or quote any external source not provided in the RAG_CONTEXT section. If regulatory text is needed but not in RAG_CONTEXT, state: "Refer to [Standard Name] for detailed requirements."

DI-6: Every section output MUST include an "advisory_recommendations" field.
  - Minimum 2 entries, maximum 4 entries.
  - Each entry MUST include: action, timeline, expected_impact, peer_reference.
  - If benchmark data is available from RAG context: recommendations MUST reference
    specific peer banks and quantified outcomes.
  - If benchmark data is NOT available: recommendations MUST be based on
    industry-standard practices (e.g., SBTi, RE100, TCFD) with clearly stated
    assumptions. Mark these as "General Best Practice" in the peer_reference field.
  - NEVER fabricate specific bank names or numbers without RAG source.
